## 详细学习指南

### 第一阶段：理解基本概念

#### 📌 第1部分：panic! - 不可恢复错误

**学习目标**
- 理解什么情况下程序会 panic
- 知道 panic 会发生什么（栈展开）
- 了解何时应该使用 panic

**核心概念**
```
panic! = 程序遇到无法处理的问题，立即终止
```

**学习要点**
1. 直接调用 `panic!` 宏
2. 数组越界等自动触发的 panic
3. panic 时的栈展开行为

**对应代码**
- 文件：`src/panic_demo.rs`
- 运行：`cargo run`

**练习建议**
- 尝试取消注释中的 panic 代码，观察程序崩溃
- 思考：什么情况下你的程序应该 panic？

**预计学习时间**：15-20 分钟

---

#### 📌 第2部分：Result<T, E> - 可恢复错误

**学习目标**
- 理解 Result 枚举的结构
- 掌握使用 match 处理 Result
- 了解 if let 的简化写法

**核心概念**
```rust
enum Result<T, E> {
    Ok(T),   // 成功，包含返回值
    Err(E),  // 失败，包含错误信息
}
```

**学习要点**
1. Result 的两个变体：Ok 和 Err
2. 使用 match 分别处理成功和失败
3. 使用 if let 简化代码（只关心一种情况时）

**对应代码**
- 文件：`src/result_basic.rs`
- 函数：`demo()`, `demo_if_let()`, `demo_match_guard()`

**练习建议**
- 编写一个函数返回 `Result<i32, String>`
- 尝试使用 match 和 if let 两种方式处理结果

**预计学习时间**：20-30 分钟

### 第二阶段：掌握实用技巧

#### 📌 第3部分：文件操作中的错误处理

**学习目标**
- 掌握文件操作中处理错误的实际方法
- 了解 unwrap() 和 expect() 的使用场景

**核心概念**

File::open("file.txt")  // 返回 Result<File, io::Error>

**学习要点**
1. 使用 match 处理文件打开结果
2. unwrap() / expect()：快速获取值（失败会 panic）
3. error.kind()：区分不同类型的错误

**对应代码**
- 文件：`src/file_handling.rs`

**练习建议**

**预计学习时间**：20-30 分钟

#### 📌 第4部分：? 运算符 - 错误传播

**学习目标**
- 理解 ? 运算符的工作原理
- 掌握 ? 在 Result 和 Option 中的使用
- 学会用 ? 简化错误传播代码

**核心概念**
```rust
let value = some_operation()?;  // 成功则继续，失败则提前返回 Err
```

**学习要点**
1. ? 运算符的本质：match 的语法糖
2. 只能在返回 Result/Option 的函数中使用
3. 链式调用多个 ?

**对比理解**
```rust
// 使用 match（繁琐）
let mut file = match File::open("file.txt") {
    Ok(f) => f,
    Err(e) => return Err(e),
};

// 使用 ?（简洁）
let mut file = File::open("file.txt")?;
```

**对应代码**
- 文件：`src/question_operator.rs`

**练习建议**
- 重写第3部分的文件操作代码，使用 ? 运算符
- 体会代码简洁性的提升

**预计学习时间**：25-35 分钟

#### 📌 第5部分：From trait - 错误类型转换

**学习目标**
- 理解 ? 运算符如何自动转换错误类型
- 学会为自定义错误类型实现 From trait
- 掌握统一错误类型的设计模式

**核心概念**
```rust
// ? 运算符会自动调用 From::from() 转换错误类型
impl From<io::Error> for AppError {
    fn from(error: io::Error) -> Self {
        AppError::Io(error)
    }
}
```

**学习要点**
1. From trait 的作用：类型自动转换
2. 定义统一的错误枚举类型
3. 实现多个 From 来支持不同错误源

**对应代码**
- 文件：`src/error_conversion.rs`

**练习建议**
- 创建一个包含 3 种错误源的自定义错误类型
- 编写函数使用 ? 同时处理这 3 种错误

**预计学习时间**：30-40 分钟

#### 📌 第7部分：Option vs Result

**学习目标**
- 明确 Option 和 Result 的使用场景
- 掌握两者之间的转换方法
- 理解何时使用哪个更合适

**对比表格**
| 特性 | Option<T> | Result<T, E> |
|------|-----------|--------------|
| 含义 | 有值或无值 | 成功或失败 |
| 变体 | Some / None | Ok / Err |
| 使用场景 | 缺失是正常的 | 需要知道失败原因 |
| 示例 | 查找元素 | 读取文件 |

**转换方法**
```rust
// Option -> Result
some_option.ok_or("错误信息")

// Result -> Option
some_result.ok()  // 丢弃错误信息
```

**对应代码**
- 文件：`src/option_vs_result.rs`

## 常见问题 FAQ

**Q1: unwrap() 和 expect() 什么时候可以用？**
> 原型开发、测试代码、你确定不会失败的情况下使用。生产代码建议使用 match 或 ?。

**Q2: Option 和 Result 怎么选？**
> 如果"没有值"是正常情况（如查找不存在的键），用 Option。如果需要知道"为什么失败"，用 Result。

**Q3: 什么时候应该 panic？**
> 程序处于无效状态、继续执行会导致更严重问题、这不是用户错误而是程序员错误时。

**Q4: 自定义错误类型必须实现哪些 trait？**
> 至少实现 `Debug` 和 `Display`。如果要在 ? 中使用，需要实现 `From`。完整的错误类型还应实现 `std::error::Error`。
